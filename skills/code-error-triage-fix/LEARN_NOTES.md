## 代码缺陷修复助手 (Code Error Triage) —— 戴着镣铐跳舞的排障专家

### 一、 核心架构与底层逻辑
**1. 痛点与绝对边界**
日常开发中，当我们把一长串报错日志扔给 AI 时，它经常会给出“建议检查网络”、“建议重启”这种毫无营养的废话，或者直接洋洋洒洒重写几百行代码。
*   **输入**：异常日志、环境上下文、有缺陷的代码片段。
*   **边界约束**：**绝对禁止**回答通用的编程概念（如“什么是闭包”），**绝对禁止**脱离当前报错去进行大规模的重构。

**2. 状态机与工作流设计**
我将排障过程抽象为了一个强顺序性的流水线：
*   **Step 1：特征提取 (Symptom Extraction)** -> 提取报错的行号、异常类名（如 `NullReferenceException`）、核心变量。
*   **Step 2：病理推断 (Root Cause Analysis)** -> 按可能性从高到低排列 3 种可能的错因。
*   **Step 3：靶向治疗 (Targeted Fix)** -> 只输出修改的那几行代码，并用 `+` 和 `-` 标出 Diff。

**3. 关键代码解析：攻击性路由**
在 `description` 中，必须精准埋点触发词。
```yaml
description: "当用户输入包含 'Traceback', 'ERR_CONNECTION_REFUSED', '未定义', '单测挂了', '缩进报错', '帮我看报错' 等词汇，或直接粘贴带有 error/exception 字样的日志时，必须强制唤醒本技能。"
```

### 二、 实战演示与场景穿透
**场景 A：复杂网络配置排障**
在配置 VS Code 代理或抓包环境时，最头疼的就是鉴权与规则冲突。
*   **User输入**：“我已经处于 rules（规则）模式了，相关的 proxy-groups 也配了，但依然无法登录对应的账号，一直报 Timeout，这是我的 config.yaml 和终端日志。”
*   **Skill执行过程**：AI 越过了“请检查是否开启规则模式”这种废话（因为我们在输入里已经界定了状态），直接定位到日志中的证书校验（如 `tls-skip-verify`）或特定的域名匹配规则（Rule Providers）被拦截的问题，精准给出需要修改的那一行 YAML 规则。

**场景 B：前端动态样式踩坑**
*   **User输入**：“我在写一个博客项目，需要文字元素根据系统的暗黑模式（Dark Mode）动态适配颜色。但是切换系统主题后，部分组件文字依然是黑色的，这是对应的 CSS 和 JS 监听代码。”
*   **Skill执行过程**：它会立刻按优先级排查：1. CSS 变量 (`var(--text-color)`) 的作用域层级是否被强样式覆盖；2. JS 中的 `matchMedia('(prefers-color-scheme: dark)')` 监听器是否正确挂载并在切换时触发了回调。

### 三、 深度心得
**防御性 AI 设计**是这里的精髓。遇到拿不准的底层框架 Bug，Skill 必须学会说“无法完全确认，建议排查 X 和 Y”，而不是强行给出一个会引发二次崩溃的错误代码。这是一种极具工程素养的表现。
